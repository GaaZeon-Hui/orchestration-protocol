# 新架构实施计划

当前代码库 `GaaZeon-Hui/orchestration-protocol@030949d`。以下分阶段实施。

---

## Phase 1: pipeline.py 状态机改造

**目标**：新的 stage 命名 + round 计数器 + ROLE_PERMISSIONS + ALLOWED_COLUMNS

| 任务 | 说明 |
|------|------|
| 替换 `VALID_TRANSITIONS` | 10 个旧 stage → 7 个新 stage（init / orchestrator_gate / worker_modify / reviewer_check / orchestrator_arbiter / verified / lock_released） |
| 替换 `ROLE_PERMISSIONS` | 新增 `reviewer` 角色；Orch 加 `worker_modify` 和 `verified` |
| 替换 `ALLOWED_COLUMNS` | 移除旧字段（conflict_analysis_json 等），新增 plan_r2-r4 / completion_r1-r4 / feedback_r1-r4 / human_intervention / review_round |
| 新增 `review_round` 列 | pipeline 表 ALTER TABLE ADD COLUMN |
| 保留 `rejected` | 作为终态之一，保持不变 |

**不动**：`transition_stage()` 核心逻辑（CAS + 白名单过滤 + 审计日志写入）结构不变，只换常量。

---

## Phase 2: orchestrator.py 数据访问层改造

**目标**：新表 + 新列 + Reviewer 支持

| 任务 | 说明 |
|------|------|
| 新增 `project` 表 | id / 项目内容 / 文件索引地址 / 改动时间 / 文件使用 / agent_status，权限规则写入 docstring |
| 新增 `register` 表 | ID / role / schema，替换 context 中的角色管理 |
| 废弃 `context` 表 | 数据迁移到 project + register 后移除 |
| `pipeline` 表新列 | plan_r2-r4 / completion_r1-r4 / feedback_r1-r4 / human_intervention / review_round 的 migrate |
| `init_pipeline()` | 适配新列名（reason_json / plan_json） |
| `get_pipeline()` | JSON 反序列化列表加新列 |
| `resolve_orphan_locks()` | 适配新 stage 名 + Orch 有 `verified` 权限 |

---

## Phase 3: lint.py 改造

**目标**：双入口，支持 plan_json 输入

| 任务 | 说明 |
|------|------|
| 新增 `validate_plan()` | 代码断言校验 plan_json 结构（files 非空、路径合法） |
| 新增 `lint_plan()` | 取 plan.files → `run_lint()`，给 orchestrator_gate 用 |
| 保留 `lint_changed_files()` | 给 reviewer_check 用，交叉验证 git diff |
| `run_lint()` | 不变（越界 glob 匹配逻辑零改动） |

---

## Phase 4: SKILL.md 角色文件

**目标**：三个角色的 LLM 行为指令

| 任务 | 说明 |
|------|------|
| 更新 `orchestrator-role/SKILL.md` | 准入 = lint_plan + reason 评估；仲裁 = 读 completion + 写 feedback 或推进 verified；heartbeat 加 Reviewer 超时扫描；人工介入流程 |
| 更新 `worker-role/SKILL.md` | 前置准备 = 读 project + register → 写 reason + plan；修正回路 = 读 feedback → 写 plan_rN → 改码；释锁 = 读 decision → 清占用 |
| 新增 `reviewer-role/SKILL.md` | 审查 = git diff + plan 交叉验证 → 写 completion + project 地址；超时检测 = /loop 扫描 → agent_status → git checkout → audit_log |

---

## Phase 5: 测试

| 任务 | 说明 |
|------|------|
| `test_pipeline.py` | 新 VALID_TRANSITIONS 全路径测试；修正回路 round 递增测试；权限拒绝测试（Worker 不可推 orch_arbiter 等） |
| `test_orchestrator.py` | init_pipeline 新格式；project 表 CRUD 权限；register 表读写；resolve_orphan_locks 新路径 |
| `test_lint.py` | validate_plan 边界 case（空 files、非法路径、非 JSON）；lint_plan 越界命中；lint_changed_files 回归 |

---

## Phase 6: 数据迁移

| 任务 | 说明 |
|------|------|
| `migrate()` 升级 | 旧 pipeline_state 行 → 符合新 schema 的 migrate 脚本；context 数据 → project + register |
| 向后兼容 | 废弃表保留但不再被新代码引用 |

---

## 实施顺序

```
Phase 1 (pipeline.py) ──▶ Phase 2 (orchestrator.py) ──▶ Phase 3 (lint.py)
                                                                │
                                                                ▼
                                                       Phase 4 (SKILL.md)
                                                                │
                                                                ▼
                                                       Phase 5 (测试)
                                                                │
                                                                ▼
                                                       Phase 6 (数据迁移)
```

Phase 1-3 是纯代码层，可以同一个人做。Phase 4 是 LLM 行为指令，依赖 Phase 1-3 的列名/函数名确定后再写。Phase 5 贯穿全程（每做完一个 Phase 就跑测试）。Phase 6 最后。
