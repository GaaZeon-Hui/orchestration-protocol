三、三种写入模式分类

模式 A：纯 SQL 状态传递（3 处）

只推进 stage 和 revision（CAS +1），不写入任何 JSON 数据列。审计日志记录 stage 变更但不含 payload。

Transition角色为什么不需要 JSON？
approved → modifying Worker granted_scope_json 已在上一步写入，Worker 读它即可知道能改什么
completion_submitted → completed Orchestrator self_review_json + commits_json 已在上一步写入，Orch 对照 git diff 验证后只推进 stage
completed → lock_released Worker 所有信息已齐全，只需释放锁标记流程结束

🔑 关键洞察：三个纯 SQL 推进恰好是 权限交接点——Orch→Worker、Worker→Orch、Worker→终态。不需要新信息，只需要"我确认上一步完成了"。

模式 B：SQL 状态推进 + JSON 传信息（6 处）

推进 stage 的同时，把自己的分析结论/产出 序列化为 JSON 写入指定列，供另一方后续读取。

一方写入，另一方读取。写和推进耦合在同一事务中（pipeline.py:142-168），读是解耦的（随时通过 orc.get_pipeline() 查询）。

模式 C：双边 JSON 接力（3 条链路）

同一语义域被两个角色用不同的列、不同的 JSON 结构接力传递：

链路Worker 写入Orchestrator 写入结构差异
Scope 链 scope_json = {"modules":[], "files":[], "excluded":[]} granted_scope_json = {"files":[], "forbidden":[]} Worker 用 modules/files/excluded 三字段申请；Orch 用 files/forbidden 两字段签发。结构不对称。
Self-Review 链 self_review_json (init) = {"potential_issues":[]} — Worker 写两次：init 时写预判，final 时覆盖写完整自审。init 版本永久丢失。
Completion 链 commits_json + context_updates_json —（Orch 只读不写） Worker 提供 commit 清单和上下文变更，Orch 对照 git diff 逐条验证。

六、四大核心机制深度解析

🔒 约束机制 (Constraints)

三层约束叠加：

第 1 层 — 模块边界 (Boundaries)

存储在 context.boundaries_json，由用户首次配置，所有 Agent 共享。
格式：{"py-agent":{"can_touch":["*.py"],"forbidden":["*.md","app/"]}}

第 2 层 — Lint 程序化校验

lint.py:_check_boundaries() — 对 git diff 的每个文件做 glob 匹配：
• 命中 forbidden → 直接 reject，不消耗 LLM
• 不在 can_touch → 直接 reject
• Lint 过 → 进入 LLM 分析

第 3 层 — LLM 约束声明

constraints_json — Worker LLM 在 init 时自报约束（如 "不可修改 public API 签名"）， Orch LLM 和 Worker 自审时对照检查。 

约束生效链路：
context.boundaries_json ──▶ lint.py 程序校验 ──▶ Orch LLM 边界分析
                                │                        │
                          不过→直接reject         边缘case→写入
                          (零LLM开销)           boundary_analysis_json

constraints_json ──▶ Orch LLM 冲突+逻辑分析 ──▶ Worker 自审对照

✅ 审批机制 (Approval)

Lint 前置 + 三项 LLM 分析 + 签发：

Step 1: lint_changed_files() 程序化预检 → 不过则直接 reject

Step 2: Orch LLM 冲突分析 → conflict_analysis_json → 推进 stage

Step 3: Orch LLM 边界分析 → boundary_analysis_json → 推进 stage

Step 4: Orch LLM 逻辑分析 → logic_analysis_json → 推进 stage

Step 5: 签发 approved（写入 granted_scope_json）或 rejected 

approval_status = "approved"
granted_scope_json = {"files":["a.py","b.py"],"forbidden":["core/"]}
reviewed_by = "orchestrator"

→ Worker 读取 granted_scope_json 作为修改边界
→ Worker 只能修改 files 白名单中的文件
→ forbidden 为黑名单补充

⚡ Orch 每步分析前先读 orc.get_pipeline(req_id) 获取最新 revision，推进后用返回值更新 revision——读-改-写循环。

🔍 自审机制 (Self-Review)

Worker 修改代码后，LLM 对照边界和约束做自我审查：

pipeline.py:115-122 强制校验：modifying → self_review_done 必须携带 self_review_json，否则抛 ValueError。 

self_review_json = {
  "files_modified": ["a.py", "b.py"],
  "files_not_in_scope": [],       ← 是否有越界修改
  "new_functions_created": ["foo","bar"],
  "breaking_changes": [],          ← 是否有破坏性变更
  "constraints_violated": [],      ← 是否违反约束
  "potential_issues": []           ← 潜在风险
}

Worker LLM 自审时必须对照 orc.get_boundaries()（模块边界）和 granted_scope_json（授权范围）。
这个 self_review_json 后续被 Orch 对照 git diff 逐条验证——不信自审，信 diff。

⚠ 注意：init 时写入的 self_review_json（预判版）被 final 时的版本覆盖，预判内容永久丢失。

🔓 释锁机制 (Lock Release)

completed 不是终态——需要 Worker 显式释锁才是真正结束：

completed ──Worker──▶ lock_released (终态)
                          │
                    transition_stage()
                    纯 SQL，无 JSON
                    revision++, stage=lock_released

为什么 completed 故意不作为终态？
这是项目 memory 中记录的刻意设计决策。如果 Orchestrator 在 completed 阶段崩溃， Worker 仍持有 active pipeline，可以恢复并从断点继续。completed 只是 "Orch 验证通过"，锁还是 Worker 的。

lock_released 之后：
• TERMINAL_STAGES = {'rejected', 'lock_released'}（pipeline.py:49）
• recover_pipeline() 忽略终态 pipeline（orchestrator.py:386-400）
• Worker 可以创建新的 pipeline（同 agent 活跃 pipeline 互斥检查只针对非终态）

八、乐观锁 (CAS) 机制：逐行拆解

8.1 什么是乐观锁？

悲观锁：读取前先锁定行（SELECT FOR UPDATE），其他人等锁释放才能读。
乐观锁（本项目的选择）：读取时不加锁，写入时检查"我读到的值是否被改了"。

选择乐观锁的原因：两个 Agent 通过 /loop 异步轮询，大部分时间没有并发写入。 WAL 模式下读者不阻塞写者，乐观锁不需要锁管理器，零争用时零开销。

8.2 CAS 三步曲（代码逐行追踪）

Step 1 — 读取当前版本号（pipeline.py:78-88）

cur.execute(
    "SELECT stage, revision FROM pipeline_state WHERE request_id=?",
    (request_id,),
)
row = cur.fetchone()
current_stage = row['stage']      # 'request_submitted'
current_revision = row['revision'] # 0

这一步不加锁。SQLite 在 WAL 模式下返回的是最后一次 commit 的快照。

Step 2 — 比对调用者持有的版本号（pipeline.py:98-104）

if current_revision != revision:   # 数据库的 0 != 调用者传入的 0?
    raise RuntimeError(
        "Revision mismatch: expected {}, actual {}".format(
            revision, current_revision,
        )
    )

这是 CAS 的第一道防线：调用者必须先读再传。如果调用者拿着过时的 revision 来推进，这里直接拦截。
但这道防线不能防止 TOCTOU（检查到使用的间隙）。真正的原子性保护在 Step 3。

Step 3 — 原子 UPDATE + rowcount 判决（pipeline.py:143-152）

cur.execute("BEGIN")
cur.execute(
    "UPDATE pipeline_state "
    "SET stage=?, revision=revision+1, updated_at=..., conflict_analysis_json=? "
    "WHERE request_id=? AND stage=? AND revision=?",  ← 三条件原子检查
    [new_stage, conflict_json, request_id, current_stage, current_revision],
)
if cur.rowcount == 0:
    raise RuntimeError("CAS failed: concurrent modification")  ← 乐观锁判决
cur.execute("INSERT INTO audit_log ...")
conn.commit()  ← UPDATE + audit_log 同一事务，全或无

为什么 rowcount=0 就是"有人抢先了"？
UPDATE 的 WHERE 带了 stage=? AND revision=?。如果另一个 Agent 在我读之后、我写之前已经推进了 stage，那么数据库中的 stage 和 revision 都已变化，WHERE 条件不匹配 → 影响 0 行 → rowcount=0。

这个检查是 原子性的——SQLite 的行级锁保证 UPDATE 的 WHERE 匹配和写入是一个不可分割的操作。不存在"查的时候匹配但写的时候被改"的窗口。

                                                    → ('request_submitted', 0)

t3: UPDATE ... WHERE revision=0
    → rowcount = 1 ✓ 成功
    → revision 变为 1
    → COMMIT

                                                t4: UPDATE ... WHERE revision=0
                                                    → rowcount = 0 ✗ CAS 失败！
                                                    → ROLLBACK
                                                    → 抛出 RuntimeError

                                                t5: 重新 SELECT → revision=1
                                                    用 revision=1 重试 → 成功

结果: 两次推进都成功，revision 最终=2。没有丢失更新，没有覆盖。
     这是测试 test_pipeline.py:219-245 验证的并发场景。

8.4 SQL Trigger 兜底

代码：orchestrator.py:126-147

即使有人绕过 pipeline.py 直接执行裸 SQL，SQLite trigger 仍然会拦截：

-- 试图跳过 revision（不 +1）
UPDATE pipeline_state SET stage='approved', revision=5
WHERE request_id='...';
→ ABORT: 'CAS failure: revision must increment by 1'

-- 试图非法转移（跳过中间 stage）
UPDATE pipeline_state SET stage='approved', revision=revision+1
WHERE request_id='...' AND stage='request_submitted';
→ ABORT: 'Invalid transition: request_submitted -> approved'
  (正确路径必须经过 conflict → boundary → logic 才能到 approved)

Trigger 定义在 BEFORE UPDATE ON pipeline_state WHEN OLD.stage != NEW.stage。
只在 stage 变化时触发，同 stage 更新其他字段不受限制。
两个 CASE WHEN 串联——第一个检查 revision 递增，第二个检查转移路径白名单。

九、WAL 模式：并发读写分离原理

9.1 WAL 是什么

WAL = Write-Ahead Logging。SQLite 默认是 rollback journal 模式（DELETE），WAL 是替代方案。

默认 (DELETE Journal)WAL (本项目)
写入方式 直接写主 DB 文件，旧页备份到 journal 新数据追加写入 WAL 文件（-wal），不碰主 DB
读者数量 1 读者 或 1 写者（互斥） 无限读者 + 1 写者（不互斥）
读-写并发 写者阻塞所有读者 写者不阻塞读者，读者不阻塞写者
写-写并发 串行（第二个等 busy_timeout） 串行（同上）
文件 xxx.db + xxx.db-journal xxx.db + xxx.db-wal + xxx.db-shm

为什么 WAL 对本项目至关重要：
Orchestrator 通过 /loop 60s 定时 invoke，每次调用可能执行多次 transition_stage()（连续写入）。
Worker 同时通过 /loop 30s 轮询 get_pipeline()（SELECT 读取）。
没有 WAL：Worker 的 SELECT 会被 Orchestrator 的 UPDATE 事务阻塞，轮询延迟不可预测。
有 WAL：Worker 读到最后一次 commit 的快照，不受 Orch 写入影响，30s 轮询节奏稳定。

9.2 WAL 在本项目中的配置

-- orchestrator.py:43-44 (_connect 方法，每次连接都执行)
PRAGMA journal_mode=WAL;       -- 启用 WAL 模式
PRAGMA busy_timeout=5000;      -- 写锁等待 5 秒（仅写-写冲突时生效）

-- orchestrator.py:52-53 (init_db)
PRAGMA synchronous=NORMAL;     -- WAL 模式下 NORMAL 足够安全，比 FULL 快很多
PRAGMA foreign_keys=ON;        -- 启用外键约束检查

9.3 WAL 的三种读操作模式

操作代码位置WAL 行为
只读查询

get_boundaries()		 orchestrator.py:397-451
orchestrator.py:471-479		 读最后 commit 的快照。即使 Orch 正在写事务中，Worker 读到的是写之前的完整状态。WAL 保证读到的是 一致快照——不会读到半更新的行。

9.4 WAL 文件管理与 Checkpoint

WAL 文件会随着写入无限增长。SQLite 在 WAL 文件达到 1000 页（约 4MB）时自动执行 checkpoint——将 WAL 内容合并回主 DB。

本项目在 orchestrator-role/SKILL.md:283 记录了手动 checkpoint 命令：PRAGMA wal_checkpoint(TRUNCATE)——用于异常情况下 WAL 文件过大的清理。

十、SQLite 功能边界：它做了什么，没做什么

SQLite 负责的部分

持久化存储三张表：pipeline_state（状态）、context（配置+心跳）、audit_log（审计）
原子性BEGIN + COMMIT / ROLLBACK 保证 UPDATE + INSERT audit_log 全或无
CAS 原子检查UPDATE WHERE revision=? 的条件匹配 + rowcount 判决是原子操作
WAL 读写分离写者不阻塞读者，支持 Worker 轮询 + Orch 写入并发
Trigger 兜底tr_stage_transition：即使裸 SQL 也无法非法转移或 revision 跳号
CHECK 约束stage 列只接受 11 个合法值
PRIMARY KEYrequest_id 唯一，防止重复创建
索引加速idx_pipeline_stage（按 stage 查询）、idx_pipeline_agent（按 agent 查询）
时间戳DEFAULT datetime('now','localtime') 自动生成
写锁排队busy_timeout=5000ms，写-写冲突时第二个等待而非立即失败

SQLite 不负责的部分
（应用层 / LLM 承担）

转移路径逻辑VALID_TRANSITIONS dict 在 Python 中定义，pipeline.py:11-21。SQLite 不知道哪些转移合法——trigger 中硬编码了同样的白名单，但那只是兜底
角色权限ROLE_PERMISSIONS dict 在 Python 中定义，pipeline.py:24-38。SQLite 不知道谁是 orchestrator 谁是 worker
字段白名单ALLOWED_COLUMNS set 在 Python 中定义，pipeline.py:41-48。SQLite 接受任何列名
JSON 结构校验所有 *_json 列存储为 TEXT——SQLite 不校验 JSON 是否合法、字段是否齐全、类型是否正确。LLM 输出 key 名拼错不会报错
LLM 内容理解JSON 列中的语义（冲突分析、边界判断、逻辑变更）对 SQLite 是完全不透明的字符串
重试逻辑CAS 失败后的重试（最多 3 次）在 SKILL.md 中描述，由 Agent 自行实现，不在 pipeline.py 中
心跳超时判断90 秒超时是应用层逻辑（orchestrator.py:213），SQLite 只存储时间戳字符串
孤儿锁回收resolve_orphan_locks() 的 120 秒超时是业务决策（orchestrator.py:323-350），SQLite 不主动清理
角色注册try_register() 用 UPDATE rowcount 做 CAS 注册（orchestrator.py:229-241），竞争逻辑在应用层
Agent 活跃互斥init_pipeline() 检查同 agent 是否有活跃 pipeline——Python 层面的 SELECT + 判断 + INSERT，非 SQL 约束

边界总结

SQLite 在这里的定位是：一个提供原子性、持久性、并发安全的状态存储器。
它不参与任何决策——不判断"这个转移合法吗"，不校验"LLM 输出的 JSON 格式对吗"，不决定"谁可以推进"。

这些全部是 pipeline.py 的 Python 逻辑 + SKILL.md 的 LLM 指令 负责。
SQLite 的 Trigger 是最后一道防线——万一 Python 逻辑有 bug，裸 SQL 也无法破坏数据一致性。
WAL 是并发基础设施——确保两个 Agent 的异步轮询不会互相阻塞。

传递模式统计：
  纯 SQL 推进:  3 次 (modifying, completed, lock_released)
  SQL + JSON:   7 次 (含 init INSERT)
  其中双边接力:  3 条链路 (scope链, self_review链, completion链)

LLM 传递统计：
  Worker→Orch: 10 个字段
  Orch→Worker: 1 个字段 (granted_scope_json)

SQLite 角色：
  存储 + 原子性(CAS rowcount) + 并发安全(WAL) + 兜底(Trigger) + 索引加速
  不做：业务逻辑、JSON校验、LLM理解、重试、心跳超时、角色注册

乐观锁 = Python 读 revision → UPDATE WHERE revision=? → rowcount 判决

Trigger = 裸 SQL 也无法非法转移或 revision 跳号（数据库层兜底）
