Orchestration Protocol

专为 Claude Code 设计的零依赖多 Agent 协作编排工具。 Worker 发起修改请求 → Orchestrator 三项审查 → 签发/拒绝 → Worker 执行 → 自审 → 验证 → 释锁。 全程由 SQLite 状态机驱动，CAS 乐观锁 + WAL 并发 + Trigger 兜底。

五 SQLite 存储架构

5.1 三张表的分工

pipeline_state

状态机主表。每行 = 一个 pipeline 请求的完整生命周期。

  • request_id PRIMARY KEY
  • stage — CHECK 约束 11 个值
  • revision — CAS 计数器
  • 16 个数据列（TEXT）
  • 索引：stage, agent

写入：init_pipeline(INSERT) + transition_stage(UPDATE)

audit_log

不可变审计日志。只追加。

  • id AUTOINCREMENT
  • role — 谁推进的
  • stage_from → stage_to
  • revision_before → revision_after
  • payload_json — 过滤后的 kwargs

写入：仅 transition_stage() — init_pipeline 不产生记录

context

全局配置单例。始终只有 id=1 一行。

  • CHECK(id=1) — 强制单行
  • boundaries_json — 模块边界
  • orchestrator_id/heartbeat — 心跳
  • agent_history/warnings

写入：set_boundaries / update_context / try_register

5.2 SQL Trigger — 数据库层兜底

定义在 orchestrator.py:126-147。BEFORE UPDATE ON pipeline_state，仅在 stage 变化时触发。

检查规则违规 →
Revision 必须 +1 NEW.revision != OLD.revision + 1 RAISE(ABORT, 'CAS failure: revision must increment by 1')
转移路径必须在白名单内 硬编码 9 条合法路径的 CASE WHEN RAISE(ABORT, 'Invalid transition: X -> Y')

即使绕过 pipeline.py 直接执行裸 SQL，Trigger 也能拦截。双重保险。

5.3 CAS 乐观锁 + WAL 并发

CAS 乐观锁流程

1. SELECT stage, revision FROM pipeline_state
2. 校验：传入 revision == 数据库 revision
3. 校验：角色有权从当前 stage 推进
4. BEGIN 事务
5. UPDATE ... WHERE request_id=? AND stage=? AND revision=?
6. if rowcount == 0 → CAS 失败 → ROLLBACK → RuntimeError
7. INSERT audit_log
8. COMMIT

并发冲突时调用方重试（最多 3 次），重新读取 revision 再试

WAL 读写分离

PRAGMA journal_mode=WAL;
PRAGMA busy_timeout=5000;
PRAGMA synchronous=NORMAL;

WAL 保证：
• 写者不阻塞读者
• 读者不阻塞写者
• 多个读者可同时读
• 写者之间仍然串行（SQLite 限制）

Worker 通过 /loop 轮询 get_pipeline()
→ 读到的是当前 commit 快照
→ 不受 Orch transition_stage() 写入阻塞

六 架构分界与分工总结

pipeline.py — 规则引擎

定义"谁能做什么、什么时候能做、怎么算合法"。

  • 状态转移白名单
  • 角色权限矩阵
  • 字段白名单
  • CAS 乐观锁实现
  • 审计日志写入

纯 Python，无 IO 依赖（除 SQLite）

orchestrator.py — 数据访问层

封装所有 SQL 操作，提供面向对象的查询接口。

  • DB 生命周期（init/migrate）
  • 角色注册（心跳 CAS）
  • pipeline CRUD
  • context 读写
  • 孤儿锁回收

委托 stage 推进给 pipeline.transition_stage()

lint.py — 程序化防线

在 LLM 介入前做机械不可漏的检查。

  • 越界 glob 匹配
  • git diff 文件冲突
  • AST 符号变更
  • 阻塞级 → 直接 reject
  • 信息级 → 喂给 LLM

零 LLM 消耗，纯程序化

SQLite — 状态存储器

提供原子性、持久性、并发安全。

  • pipeline_state — 状态机主表
  • audit_log — 不可变审计
  • context — 全局配置单例
  • CAS 原子性 (rowcount)
  • WAL 读写分离
  • Trigger 兜底

不参与业务决策

SKILL.md — LLM 行为指令

告诉 LLM 在什么角色下该做什么。

  • 启动流程
  • Stage 推进顺序
  • JSON 写入格式
  • 崩溃恢复逻辑
  • /loop 轮询策略

人类可读，LLM 执行

status.py — 只读监控

观察者视角，不影响任何业务逻辑。

  • 轮询 pipeline_state
  • 检测 audit_log 变化
  • 终端可视化
  • --once 供 Agent 调用

纯只读，RO 连接打开 DB

#Stage推进者写入的数据传递模式关键校验
0request_submitted Worker reason · scope_json · plan_json · self_review_json(init) · constraints_json JSON 同 agent 无活跃 pipeline
1conflict_analysis_done Orch conflict_analysis_json JSON Lint 程序预检通过
2boundary_analysis_done Orch boundary_analysis_json JSON CAS revision 匹配
3logic_analysis_done Orch logic_analysis_json JSON CAS revision 匹配
4approved Orch granted_scope_json · approval_status · reviewed_by JSON 三项分析均通过
5modifying Worker 无 — 纯 stage 推进 纯SQL Worker 读取 granted_scope_json 后执行
6self_review_done Worker self_review_json(final) ⚠强制必填 JSON pipeline.py:115-122 非空校验
7completion_submitted Worker commits_json · sync_notes · context_updates_json JSON —
8completed Orch 无 — 纯 stage 推进 纯SQL 对照 git diff 验证 self_review_json
9lock_released ◀终态 Worker 无 — 纯 stage 推进 纯SQL —
纯SQL = 3 次推进（modifying · completed · lock_released）
JSON = 7 次写入（含 init INSERT）
