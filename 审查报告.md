# Orchestration Protocol 审查报告

2026-05-22

## 概述

对 `GaaZeon-Hui/orchestration-protocol` 仓库的实操性审查。该项目是一个基于 Claude Code skill 机制的多 agent 编排协议——Orchestrator（审查者）+ Worker（执行者）通过 SQLite 数据库 `pipeline_state` 状态机协调代码修改任务。

**审查范围**：`pipeline.py`、`orchestrator.py`、`lint.py`、3 个 SKILL.md、测试套件（72 测试/全绿）。

---

## 架构

```
CLAUDE.md (入口)
  └─ 执行 Python → 确定角色 → 读对应 SKILL.md
       ├─ orchestrator-role/SKILL.md  →  哨兵检查 → lint 预检 → 三项分析 → 审批/拒绝
       └─ worker-role/SKILL.md       →  配置边界 → init_pipeline → 等审批 → 修改 → 自审 → completion

pipeline.py     transition_stage()  CAS + 权限 + 审计（单文件，stdlib only）
orchestrator.py 查询 + 注册 + check_and_heartbeat()（单文件，stdlib only）
lint.py         越界检查 + 冲突检测 + AST 提取（单文件，stdlib only）
```

状态机：`request_submitted → conflict_analysis_done → boundary_analysis_done → logic_analysis_done → approved → modifying → self_review_done → completion_submitted → completed → lock_released`（或任一分析阶段 → rejected）

---

## 优点

### 状态机设计自洽

`pipeline.py` 的 `transition_stage()` 设计简洁有力：CAS（`WHERE stage=? AND revision=?`）+ 权限校验 + SQL trigger 兜底 + 同一事务内写入 `audit_log`。并发测试（两个线程同时推进同一 pipeline）通过，CAS 保证只有一个成功。

### 权限分离清晰

ROLE_PERMISSIONS 将 orchestrator 和 worker 的 stage 推进权限完全分离。`PermissionError` 在 Python 层和 trigger 层双重保护，测试覆盖了交叉权限拒绝。

### 零外部依赖

所有 Python 模块仅依赖 stdlib（`sqlite3`、`ast`、`fnmatch`、`subprocess`、`json`），与设计原则一致。

### lint 层是正确方向

越界检查从 LLM 自觉改为 glob match 的确定性校验，冲突检测改用 `git diff --name-only base_ref..HEAD` 文件交集。阻塞级直接 reject，信息级（AST 变更摘要）喂给 LLM 辅助逻辑分析。

### 测试覆盖到位

72 个测试覆盖：CAS 并发、权限矩阵、全流程、reject 路径、trigger 兜底、崩溃恢复、心跳超时 takeover、边界 CRUD、pattern 匹配、AST 解析/比对、lint 集成。

## 问题清单

### 状态：已解决 ✓

#### 1. Monitor 是阻塞 Python generator（已解决）

`orchestrator.py` 的 `run_monitor()` 是 `while True` + `time.sleep()` 的无限 generator。Claude Code agent 无法在后台运行持久 Python 进程，同时进行审查工作。

**解决方案**：新增 `check_and_heartbeat()` 单次检查方法。Agent 每次调用只查一次，需要持续监控时用 `/loop` 定时复invoke。SKILL.md 已同步更新。

#### 2. Worker 的轮询等待不兼容 Agent 工作流（已解决）

Worker SKILL.md Step 5 和 Step 9 包含 `while True: p = orc.get_pipeline(req_id) ... time.sleep(2)`。

**解决方案**：全部改为 check-once 模式。未审批时输出建议 `/loop 30s` 后 return。下次 `/loop` invoke 时从 Step 1（崩溃恢复）进入，`recover_pipeline()` 定位断点后继续。

#### 3. 三项分析全靠 LLM 自觉（已解决）

冲突分析、越界分析、逻辑分析全是 markdown 里写给 LLM 的指令，没有程序化校验。

**解决方案**：新增 `lint.py`，越界检查（glob match，阻塞级）、文件冲突检测（git diff 交集，信息级）、AST 变更提取（函数签名、新增/删除符号、新类，信息级）。阻塞级直接 reject 不消耗 LLM context，信息级作为结构化输入喂给 LLM。

---

### 状态：部分缓解 ~

#### 4. Orchestrator 崩溃恢复没有分析中间态（部分缓解）

`pipeline_state` 只存 stage，不存分析中间结果。旧 orch 崩溃后，新 orch 需要重做三项分析。

**缓解**：lint 检查是纯机械的（秒级重跑），不影响正确性。但 LLM 分析结论仍然丢失，新 orch 需要重新审查。

**剩余问题**：若旧 orch 已完成了 conflict 和 boundary 分析（LLM 判断通过），新 orch 重做可能得出不同结论——模型不确定性。

#### 5. Worker 修改阶段没有实时变更校验（部分缓解）

Worker 可以 `transition_stage(req_id, 'modifying', ...)` 后凭空 `transition_stage(req_id, 'self_review_done', ...)` 而不做任何实际修改。

**缓解**：Orchestrator 验证 completion 时有了 lint 的结构化 hints 做对照，比纯肉眼对照 `git diff` 可靠。

**剩余问题**：修改阶段本身没有实时校验。Worker 仍然可以声称"改了 a.py"但实际没改，或者在修改阶段触碰了未授权文件。Orchestrator 事后对照可以发现问题，但无法预防。

### 状态：未解决 ✗

#### 6. 引导悖论（P0 — 致命）

CLAUDE.md 写的是"启动时执行 Python 代码确定角色"，但：

- Claude Code 加载 CLAUDE.md 时不会自动执行 Python。它只是把指令读进去。
- 入口 skill `orchestration-protocol/SKILL.md` 描述了角色注册流程，但**没有写 `Skill(skill="orchestrator-role")` 或 `Skill(skill="worker-role")` 的显式 invoke 逻辑**。
- Agent 读到入口 skill 后可能输出"ROLE: worker"然后停住，不知道下一步该 invoke 哪个角色 skill。

**影响**：整套系统能否在实际 Claude Code session 里跑起来的第一道关卡未解决。前面的改进都是在优化一个实际上可能启动不了的流程。

**修复方向**：在入口 SKILL.md 中明确写死——角色判定后立即调用 `Skill(skill="orchestrator-role")` 或 `Skill(skill="worker-role")`。

---

#### 7. `migrate()` 吞掉所有异常（P1）

```python
for col in ["orchestrator_id", "orchestrator_heartbeat",
             "orchestrator_started_at", "boundaries_json"]:
    try:
        conn.execute("ALTER TABLE context ADD COLUMN {} TEXT".format(col))
    except sqlite3.OperationalError:
        pass
```

`sqlite3.OperationalError` 不止是"列已存在"——还可能是数据库被锁定、磁盘满、权限错误。这些全被静默吞掉。

**修复方向**：检查错误信息是否为 "duplicate column name"，否则重新抛出。

---

#### 8. 两个同类 Worker 可以并发创建冲突 request（P1）

Step 3 只检查同 agent 是否有 `modifying` 状态的 pipeline。两个 worker（同 agent 类型）可以在各自 session 里同时 `init_pipeline()`，都处于 `request_submitted`。

Orchestrator 审批第一个时不知道第二个的存在，可能审批两个互相冲突的修改——第二个 pipeline 基于审批前的代码状态，第一个执行后第二个的 base 已经变了。

**修复方向**：`init_pipeline()` 前检查同 agent 是否已有任何非终态 pipeline（不只是 `modifying`）。

---

#### 9. Timestamp 格式比较在 `run_monitor()` 中不一致（P3 — 影响范围已缩小）

```python
_ts_fmt = '%Y-%m-%d %H:%M:%S.%f'
now_check = datetime.now().strftime(_ts_fmt)
# vs
# updated_at 是 SQLite datetime('now','localtime') — 无微秒
```

字典序比较 `'2026-05-22 15:00:00' > '2026-05-22 15:00:00.000000'` 存在边界问题。

**影响**：`run_monitor()` 已被 `check_and_heartbeat()` 替代为主要路径，影响缩小。但方法本身仍在代码中，如被调用仍有隐患。

---

#### 10. 单 SQLite 文件 = 单机限制（P3 — 设计约束，非缺陷）

所有 agent 共享同一个 `orchestrator.db` 文件。WAL 模式支持并发读+单写，在多 session 本地跑可行。但不能跨机器。

**影响**：当前使用场景是单用户多 session，够用。SQLite → PostgreSQL 迁移已在 roadmap 中，待多人多机场景触发。

## 与前版对比

| 维度 | 审查前 | 审查后 |

| 越界检查 | LLM 自觉 | lint.py glob match（阻塞级） |
