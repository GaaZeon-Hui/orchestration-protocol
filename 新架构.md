# 异步流水线与配置状态机架构设计

本架构采用 **SQLite 本地表** 存储状态，通过 **Orchestrator**、**Worker** 和 **Reviewer** 三方协作，推进带命名的 `stage` 在 7 个主阶段之间流转。修正回路通过 `review_round` 计数器管理，不增加 stage 数量。

---

## 1. 角色职责三角

```
          Worker ──"我要改这些，按这个计划"──▶ Orchestrator
                                                  │
          Worker ◀──"批准，范围是这些"─────────────┘
            │
            │ 改代码，commit
            ▼
          Reviewer ──"commit 是否 = plan 声明"──▶ completion 结论
                                                      │
          Orchestrator ◀──"Reviewer 的结论"──────────┘
            │
            │ 仲裁：通过 → verified → 释锁
            │       不通过 → feedback → 修正回路
            ▼
          Worker
```

| 角色 | 核心职责 | 一句话 |
|------|---------|--------|
| **Worker** | 前置准备 → 声明计划 → 改码 → 自纠 | "我知道要改什么，我按计划改" |
| **Orchestrator** | 准入审批 → 仲裁 → 项目管理 → 人工介入协调 | "我管全局，我不亲自验代码" |
| **Reviewer** | 只做一件事：验证 commit 是否符合 plan | "你说你改了这些——我核对事实" |

## 2. 设计约束准则

### 2.1 信息简洁性原则

| 场景 | 原则 |
|------|------|
| **Agent 间协议** | 信号字段只传必要结构化数据，零冗余。同一个信息不重复出现在两个字段中。 |
| **人类界面** | human_intervention 对话、status.py 面板、Orch 汇报 —— 才增加可读性。 |

- completion_rN：只描述"多改了什么、少改了什么"，不附带文件路径、行号、上下文推测。一步到位。
- feedback_rN：只描述"需要修正什么"，不解释为什么 Review 没通过（Worker 不需要知道 Review 的推理链）。
- plan_rN：只声明"改什么文件、怎么改"，不写动机（动机在 reason_json 中）。
- reason_json：只写改哪块功能 + 为什么 + ID + schema。Orch 读一次即够，不需要嵌套引用。

**遵守本原则的场景**：所有 role 之间的 JSON 字段内容，SKILL.md 指令设计。

### 2.2 列权限最小化原则

每个角色只读自己需要的列，只写自己负责的列。跨越读取其他角色的推理过程是禁止的。

**已定义边界**：
- Worker 不能读 Reviewer 的 completion 原文（通过 Orch feedback 理解修正方向）
- Reviewer 不能读 Orch 的内部记忆或 human_intervention
- Orch 不能读 Worker 的 commit diff 直接验证（那是 Reviewer 的职责）

### 2.3 人类不编辑表

用户通过对话告诉 Orch，Orch 亲手推进。用户永远不直接操作 SQLite 表。

### 2.4 一列一责

每个 JSON 列只承载一个角色的产出，不混用。同一语义域的多轮产物通过 `_rN` 后缀区分（如 `plan_r2`），而非复用同一列。

### 2.5 终态不可逆

`rejected`、`lock_released` 是终态。进入终态的 pipeline 不再被 `recover_pipeline()` 捞起。新 pipeline 只能通过新 `init_pipeline()` 创建。

## 3. 数据存储层 (SQLite Tables)

### 3.1 四表总览

| 表名 | 职责 | 关键特征 |
|------|------|---------|
| **pipeline** | 流水线状态主表 | 预定义列 + 白名单过滤；4 轮修正预留列 |
| **project** | 项目索引 + 文件占用 + Agent 状态 | 文件索引地址作为冲突检测中枢 |
| **register** | Agent 角色注册 + schema | 各 Agent 启动时查自己的 schema |
| **audit_log** | 不可变审计日志 | transition_stage 自动写入；Reviewer 复原文件时直接 INSERT |

### 3.2 底层并发安全机制

- **CAS (Compare-And-Swap)**: `transition_stage()` 更新 `stage` 和 `revision` 时使用 `WHERE revision=?` + `rowcount` 判决
- **WAL (Write-Ahead Logging)**: 写者不阻塞读者——Worker/Orch/Reviewer 的 `/loop` 轮询互不干扰
- **SQL Trigger**: BEFORE UPDATE ON pipeline，stage 变化时校验 revision 必须 +1 且转移路径在白名单内

### 3.3 表 Schema 详情

#### pipeline 表

```
核心列：
  stage              TEXT    — 状态标识（CAS 锁定列）
  revision           INT     — CAS 版本计数器
  review_round       INT     — 修正轮次（默认 1，Orch 写）
  agent              TEXT    — Worker 标识

Worker init 写入（原始计划）：
  reason_json        TEXT    — 改动原因 + Worker ID + schema 元数据
  plan_json          TEXT    — 原始计划（无后缀，区别于修正计划）

Worker 修正写入（按轮次）：
  plan_r2            TEXT    — 第 2 轮修正计划
  plan_r3            TEXT    — 第 3 轮修正计划
  plan_r4            TEXT    — 第 4 轮修正计划

Worker 执行写入：
  commits_json       TEXT    — commit hash 列表

Orchestrator 准入写入：
  approval_status    TEXT    — 'approved' 或 'rejected'
  rejection_reason   TEXT    — 拒绝原因（Lint 不过 或 Orch 评估后否决）

Orchestrator 仲裁写入（按轮次）：
  feedback_r1        TEXT    — 第 1 轮修正指令（给 Worker）
  feedback_r2        TEXT    — 第 2 轮修正指令
  feedback_r3        TEXT    — 第 3 轮修正指令
  feedback_r4        TEXT    — 第 4 轮修正指令

Reviewer 审查写入（按轮次）：
  completion_r1      TEXT    — 结构化结论（多改了什么、少改了什么）
  completion_r2      TEXT    — 第 2 轮
  completion_r3      TEXT    — 第 3 轮
  completion_r4      TEXT    — 第 4 轮

Orchestrator 人工介入（仅 Orch 可读写）：
  human_intervention TEXT    — JSON: needs_human / decision / 用户对话 / Orch 简略回复 / 时间 / 项目名

Worker 只读 human_intervention 中的 decision 字段（'continue' 或 'reject_restart'）
```

#### project 表

```
  id               TEXT    — Orch 开新行时写入
  项目内容          TEXT    — Orch 写入
  文件索引地址      TEXT    — Reviewer 审完后写入
  改动时间          TEXT    — Reviewer 审完后写入
  文件使用          TEXT    — Worker 占用（写入自己 id）/ 释放（清除）
  agent_status     TEXT    — Reviewer 写入（'异常' / '异常还原' / NULL 清除）

权限规则：
  Orch:     W: id + 项目内容
  Worker:   R: 全部  W: 仅 文件使用
  Reviewer: R: 全部  W: 文件索引地址 + 改动时间 + agent_status
```

#### register 表

```
  ID               TEXT    — Agent 唯一标识
  role             TEXT    — 'worker' | 'orchestrator' | 'reviewer'
  schema           TEXT    — 该 Agent 的校验规则 / 注入信息 (JSON)
```

**schema JSON 结构**（单 agent 视角，每行只描述自己）：

```json
{
  "can_touch":   ["*.py", "lib/**"],
  "forbidden":   ["core/__init__.py", "*.md", "app/"],
  "constraints": ["不可修改 public API 签名", "需保持 Python 3.10+ 兼容"]
}
```

Worker 前置准备时查 register 表 → 获取自己的 schema → 注入 reason_json 的元数据中。Lint 在 `orchestrator_gate` 读 plan_json 的 `files` 列表时，用 schema 中的 `can_touch` / `forbidden` 做越界 glob 匹配。

## 4. 阶段状态机 (Stage State Machine)

### 4.1 转移图

```
                    ┌─────────┐
                    │  init   │  Worker 发起
                    └────┬────┘
                         │
                    ┌────▼──────────┐
                    │orchestrator_  │  Orch 准入 (Lint + reason 评估)
                    │    gate       │
                    └────┬────┬─────┘
                         │    │
              ┌──────────┘    └──────────┐
              ▼                          ▼
       ┌──────────────┐           ┌──────────┐
       │worker_modify │           │ rejected │ ← 终态
       └──────┬───────┘           └──────────┘
              │
       ┌──────▼───────┐
       │reviewer_check│  Reviewer 审查
       └──────┬───────┘
              │
       ┌──────▼──────────┐
       │orchestrator_    │  Orch 仲裁
       │   arbiter       │◄───────────────┐
       └──────┬────┬─────┘                │
              │    │                      │
    ┌─────────┘    └──────────┐           │
    ▼                         ▼           │
┌──────────┐           ┌──────────────┐   │
│ verified │           │worker_modify │───┘  (review_round < 4 时)
└────┬─────┘           │  (修正回路)   │
     │                 └──────────────┘
     ▼
┌───────────────┐
│lock_released  │ ← 终态
└───────────────┘
```

### 4.2 形式化定义

```python
VALID_TRANSITIONS = {
    'init':                 ['orchestrator_gate'],
    'orchestrator_gate':    ['worker_modify', 'rejected'],
    'worker_modify':        ['reviewer_check'],
    'reviewer_check':       ['orchestrator_arbiter'],
    'orchestrator_arbiter': ['verified', 'worker_modify'],
    'verified':             ['lock_released'],
}

# 终态：rejected, lock_released
# needs_human 不是 stage，是 human_intervention 列的标记
```

### 4.3 ROLE_PERMISSIONS

```python
ROLE_PERMISSIONS = {
    'worker':        {'init', 'worker_modify', 'verified'},
    'orchestrator':  {'orchestrator_gate', 'orchestrator_arbiter', 'verified', 'worker_modify'},
    'reviewer':      {'reviewer_check'},
}
# 'verified' 由 Worker 和 Orch 共享
#   Worker:  正常释锁
#   Orch:    孤儿锁回收 (Worker 超时死亡 + Reviewer 确认 + Orch 代为释锁)
# 'worker_modify' 由 Worker 和 Orch 共享
#   Worker:  正常推进到 reviewer_check
#   Orch:    Reviewer 超时检测后，Orch 代为推进 (worker_modify → reviewer_check)

### 4.4 ALLOWED_COLUMNS（字段白名单）

```python
ALLOWED_COLUMNS = {
    # Worker init
    'reason_json', 'plan_json',
    # Worker 修正
    'plan_r2', 'plan_r3', 'plan_r4',
    # Worker 执行
    'commits_json',
    # Orch 准入
    'approval_status', 'rejection_reason',
    # Orch 仲裁
    'feedback_r1', 'feedback_r2', 'feedback_r3', 'feedback_r4',
    # Reviewer
    'completion_r1', 'completion_r2', 'completion_r3', 'completion_r4',
    # Orch 人工介入
    'human_intervention',
    # 计数器
    'review_round',
}
```

## 5. 修正回路与 round 计数器

### 5.1 机制

```
review_round 列（INT, DEFAULT 1）

Orch 仲裁为 N → 推进到 worker_modify → review_round += 1
Orch 仲裁为 Y → 推进到 verified → review_round 不变

Worker 写 plan_r{round}    （round=1 是 plan_json 即原始计划）
Orch   读 completion_r{round} / 写 feedback_r{round}
Reviewer 写 completion_r{round}
```

### 5.2 回路流程

```
第 1 轮（主路径）:
  init → orch_gate → worker_modify → reviewer_check → orch_arbiter
  → Y → verified → lock_released ✓

  → N → worker_modify（review_round=2）

第 2 轮（修正）:
  Worker 写 plan_r2 → 改码 → worker_modify→reviewer_check
  Reviewer 写 completion_r2 → orch_arbiter
  → Y → verified → lock_released ✓
  → N → worker_modify（review_round=3）

第 3 轮: 同上，plan_r3 / completion_r3 / feedback_r3

第 4 轮: 同上，plan_r4 / completion_r4 / feedback_r4
  → N → Orch 写 human_intervention（needs_human: true）
  → 不推进 stage，等待用户
```

### 5.3 轮次上限

| 条件 | 行为 |
|------|------|
| `review_round < 4` 且 Orch 倾向 N | 写 feedback_rN → 推进到 worker_modify |
| `review_round >= 4` 且 Orch 倾向 N | 写 human_intervention → 暂停 → 等用户 |
| 用户决定 "继续修正" | Orch 推进到 worker_modify（review_round 不增加，仍为 4） |
| 用户决定 "否决重建" | Orch 推进到 verified（human_intervention.decision = "reject_restart"） |
| 用户决定 "通过" | Orch 推进到 verified |

## 6. 信息流权限三角

```
                  ┌──────────┐
                  │  Worker  │
                  │改码+写plan│
                  └────┬─────┘
                       │ 推进的文件（代码变更）
                       ▼
               ┌───────────┐
               │ Reviewer  │ ← 只能读 Worker 推进的文件 + plan_rN
               │ 写completion│ ← 只能写 completion_rN + project.文件索引地址
               └─────┬─────┘
                     │ completion_rN
                     ▼
               ┌───────────┐
               │Orchestrator│ ← 只能读 Reviewer 的 completion_rN
               │写feedback  │ ← 只能写 feedback_rN 或推进到 verified
               │或 verify   │
               └─────┬─────┘
                     │ feedback_rN（修正指令）
                     ▼
               ┌──────────┐
               │  Worker  │ ← 只能读 Orch 的 feedback_rN 或 verified
               └──────────┘
```

信息单向流动。每个角色只看到上一环节的**输出列**，无法跨越读取。Worker 在修正阶段不能读到 Reviewer 的 completion 原文——只能通过 Orch 的 feedback 了解需要修正什么。

Worker 在 `lock_released` 后读取 `human_intervention` 列——但仅限 `decision` 字段（`"reject_restart"` 或 `null`）。用户对话和 Orch 内部记忆对 Worker 是噪音。

## 7. 逐 Stage 详解

### Stage 0: init（Worker 发起）

**前置准备（pipeline 创建前）：**

```
Worker 启动
  ├─ R: project 表 → 查文件索引地址
  │   └─ 已有 → 检查 文件使用 是否被占用
  │       └─ 被占用 → 等待
  │       └─ 未占用 → 文件使用 = 自己的 ID
  │   └─ 不存在 → 后续由 Lint 判断，接受一次 LLM context 浪费
  ├─ R: register 表 → 用自己的 ID 查 schema → 注入信息
  ├─ R: 项目上下文
  └─ 产出 reason_json 和 plan_json
```

**创建 pipeline：**

```
init_pipeline():
  → INSERT pipeline (stage='init', reason_json, plan_json)
  → 同 agent 活跃 pipeline 互斥检查（沿用当前 init_pipeline 逻辑）
```

### Stage 0→1: orchestrator_gate（Orch 准入）

```
Orch 在 orchestrator_gate:
  1. Lint 程序化读 plan_json → 校验 plan 结构合法性
     → 结构不合法 → 直接推进到 rejected
  2. Lint 提取 plan.files → 对照 register.schema 做越界 glob 匹配
     → 命中 forbidden 或超出 can_touch → 直接推进到 rejected（不读 reason，不消耗 LLM）
  3. Lint = 通过 → Orch LLM 读 reason_json → 评估改动合理性
     → 不合理 → 推进到 rejected
     → 合理 → 写 approval_status='approved' → 推进到 worker_modify
```

**Lint plan 结构校验**（lint.py 新增 `validate_plan()` + `lint_plan()`）：

```python
def validate_plan(plan_json_str):
    """Worker 写入的 plan_json 必须满足此结构，否则直接 reject。"""
    try:
        p = json.loads(plan_json_str)
    except (json.JSONDecodeError, TypeError):
        return False, "plan_json 不是合法 JSON"

    if not isinstance(p.get("files"), list) or len(p["files"]) == 0:
        return False, "files 缺失或为空"

    for f in p["files"]:
        if not isinstance(f, str):
            return False, "files 中有非字符串条目"
        if f.startswith("/") or ".." in f:
            return False, "文件路径越界: {}".format(f)

    return True, p


def lint_plan(plan_json_str, boundaries, agent):
    """orchestrator_gate 用：读 plan 声明的文件列表做越界检查。"""
    ok, result = validate_plan(plan_json_str)
    if not ok:
        return {"blocked": True, "reason": result, "hints": None}
    files = result["files"]
    return run_lint(files, boundaries, agent)
```

**plan_json 标准格式**：

```json
{
  "files":   ["a.py", "lib/utils.py"],
  "changes": [
    {"file": "a.py",        "type": "modify", "hint": "第42行添加None检查"},
    {"file": "lib/utils.py","type": "add",    "hint": "新增get_ordinal函数"}
  ]
}
```

`files` 必填（Lint 越界检查的输入）。`changes` 可选（信息级，供 Orch LLM 参考）。

### Stage 1→2: worker_modify（Worker 改码）

```
Worker 在 worker_modify:
  1. 检测到 approval_status='approved'
  2. 按 plan_json 中声明的范围修改代码
     （修改范围已在 register.schema 注入到 reason/plan 时确定，不另设 granted_scope 列）
  3. 写 commits_json
  4. 推进到 reviewer_check
```

### Stage 2→3: reviewer_check（Reviewer 审查）

```
Reviewer 执行顺序:
  1. 读 plan_rN（当前 round 对应计划）
  2. 定位 Worker 修改的文件:
     a) plan 中声明的 files 列表
     b) git diff --name-only（lint_changed_files 保留原有逻辑）
     → 交叉验证：plan 声明 vs 实际改动
  3. project 表 → 写入文件索引地址 + 改动时间
  4. 写 completion_rN（结构化结论，侧重多改/少改，语言精炼一步到位）
  5. 推进到 orchestrator_arbiter

completion_rN 结构:
{
  "verdict": "符合计划" | "有偏差" | "worker_timeout",
  "extra_changes": ["多改了什么 — 一行概括"],
  "missing_changes": ["少改了什么 — 一行概括"],
}
```

### Stage 3→4: orchestrator_arbiter（Orch 仲裁）

```
Orch 在 orchestrator_arbiter:
  读 completion_r{review_round}
  → 通过（verdict == "符合计划"）:
      → 推进到 verified
  → 不通过 且 review_round < 4:
      → 写 feedback_r{review_round}（修正指令）
      → review_round += 1
      → 推进到 worker_modify
  → 不通过 且 review_round >= 4:
      → 写 human_intervention: {needs_human: true, ...}
      → 暂停该 pipeline
      → status.py 面板标红
      → 等待用户

用户介入:
  → 告诉 Orch 决定
  → Orch 追加 human_intervention: {decision: "...", by: "user", ...}
  → 推进到 verified 或 worker_modify（视用户决定）
```

### Stage 4→5: verified（Worker 释锁验证）

```
Worker 在 verified:
  1. 推进到 lock_released
  2. 读 human_intervention.decision:
      → null → 流程结束
      → "reject_restart" → 清除 project.文件使用 → 新建 pipeline 重新开始
```

### Stage 5→6: lock_released（终态）

```
pipeline 进入终态。recover_pipeline() 忽略终态 pipeline。
Worker 可以创建新 pipeline。
```

## 8. Reviewer 超时检测与文件复原

### 8.1 扫描机制

```
Reviewer 通过 /loop invoke（方案 A，自身定时器）:
  → 扫描: stage='worker_modify' 且 updated_at > 120s 未更新
  → 检查 Worker 是否在最近 /loop 周期内更新过 pipeline.updated_at
      → 有更新 → Worker 活着，只是慢 → 跳过
      → 无更新 → Worker 确认死亡 → 启动复原
```

### 8.2 复原流程

```
Reviewer:
  1. project.agent_status → '异常'
  2. 定位 Worker 修改的文件
  3. git checkout -- <file>  （复原修改）
  4. 向 audit_log 直接 INSERT 一行（不依赖 transition_stage）
     内容: 复原文件列表 + 时间 + 触发原因（哪个 Worker 超时）
  5. project.agent_status → '异常还原'
  6. 写 completion_rN（推荐推进: {verdict: "worker_timeout", ...}）
  7. 推进: worker_modify → reviewer_check → orchestrator_arbiter
```

### 8.3 Orch 处理超时

**Worker 超时：**

```
Orch 在 orchestrator_arbiter 读到 completion 中有超时标记:
  → human_intervention 记录: needs_human + Worker 超时信息
  → 推进到 verified
  → 向用户汇报
  → 替 Worker 释锁（Orch 有 verified→lock_released 的权限）
```

**Reviewer 超时（Orch 心跳周期检测）：**

```
Orch 在 heartbeat 周期扫描:
  → stage='reviewer_check' 且 updated_at > 120s 未更新
  → 写 human_intervention: needs_human + Reviewer 超时信息
  → 向用户汇报，等待决策
  → 不推进 stage（Orch 不能替代 Reviewer 写 completion）
```

### 8.4 agent_status 状态机

```
         Reviewer 检测到 Worker 超时
         ┌──────────┐
         │   异常    │  ← Reviewer 写（Worker 读到 → 等待，不抢占用）
         └────┬─────┘
              │ 文件复原完成
              ▼
         ┌──────────┐
         │  异常还原  │  ← Reviewer 写（Worker 读到 → 改 文件使用=自己 ID，接管文件）
         └────┬─────┘
              │ 新 Worker 正常编辑后，Reviewer 下次来时
              ▼
         ┌──────────┐
         │  (NULL)   │  ← Reviewer 清除（表示正常）
         └──────────┘
```

### 8.5 超时 Worker 回归

```
超时 Worker 回来后:
  → 读自己的原 reason_json → 重新开始 pipeline
  → 重新: 读 project/register → 检查占用 → 写 plan → init
  → 接管文件时从零开始（不继承死前状态）
```

## 9. 人工介入机制

### 9.1 触发条件

| 条件 | 行为 |
|------|------|
| `review_round < 4` + Orch 仲裁 N + 用户主动介入 | 允许。用户可在任意轮次介入，决定继续或 verify |
| `review_round >= 4` + Orch 仲裁 N | 强制。Orch 不再推进，写 human_intervention |
| Reviewer 报告 Worker 超时 | 强制。Orch 汇报用户 |

### 9.2 human_intervention 列结构

```json
{
  "needs_human": true,
  "round": 4,
  "project_name": "...",
  "timestamp": "2026-05-27 12:00:00",
  "trigger": "max_round" | "worker_timeout" | "user_requested",
  "summary": "第 4 轮审查仍检测到多改 core/__init__.py",
  "user_conversation": [
    {"role": "orch", "msg": "pipeline xxx 第4轮仍不通过，请决策"},
    {"role": "user", "msg": "否决，让 Worker 重建 pipeline"}
  ],
  "decision": "reject_restart" | "continue_r4" | "verify",
  "decided_at": "2026-05-27 12:05:00",
  "decided_by": "user"
}
```

### 9.3 用户交互流程

```
Orch 写 human_intervention.needs_human = true
  → status.py 面板检测 → 标红显示
  → Orch 暂停处理该 pipeline
  → 用户通过对话告诉 Orch 决定
  → Orch 追加 human_intervention（decision + 对话记录）
  → Orch 亲手推进（用户不直接编辑任何表）
```

## 10. 审计日志覆盖

### 10.1 transition_stage 自动记录

所有 stage 变更自动写入 audit_log（跟当前架构一致）。记录 role / stage_from / stage_to / revision_before / revision_after / payload_json。

### 10.2 Reviewer 复原文件时的手动记录

Reviewer 在复原 Worker 修改的文件时，不依赖 transition_stage，直接 INSERT audit_log：

```
INSERT INTO audit_log
  (request_id, role, stage_from, stage_to,
   revision_before, revision_after, payload_json)
VALUES
  (req_id, 'reviewer', 'worker_modify', 'worker_modify',
   0, 0,

     "reason":"worker_timeout","worker_id":"py-agent-xxx",
     "timestamp":"2026-05-27 12:03:00"}')
```

### 10.3 不产生审计记录的操作

| 操作 | 原因 |
|------|------|
| Reviewer 定时器扫描 | 纯 SELECT |
| `git checkout -- <file>` | 外部 git 命令 |
| 修改 project.agent_status | 直接 UPDATE project 表 |
| Worker 修改 project.文件使用 | 直接 UPDATE project 表 |
| Reviewer 修改 project.文件索引地址 | 直接 UPDATE project 表 |

## 11. Worker 文件占用与释放

### 11.1 占用流程

```
Worker 前置准备:
  1. 查 project 表 → 找到目标文件索引地址
  2. 检查 文件使用 字段:
      → NULL → 写入自己的 ID → 继续
      → 被占用 → 等待（/loop 轮询检查）
  3. 同时检查 agent_status:
      → NULL → 正常，继续
      → '异常' → 等待（不在异常状态的资源上操作）
      → '异常还原' → 改 文件使用=自己 ID → 接管
```

### 11.2 释放流程

```
Worker 在 lock_released 后:
  1. 读 human_intervention.decision
  2. 查 project.文件使用 = 自己的 ID
  3. 清除 文件使用（设 NULL）
  4. decision = "reject_restart" → 新建 pipeline 重新开始
     decision = null → 流程结束
```

### 11.3 Orch 孤儿释放

```
Orch 在 Reviewer 报告 Worker 超时后:
  1. 推进 verified → lock_released
  2. 不需要清除 project.文件使用（后续接管 Worker 会覆盖）
```

## 12. 与当前架构的关键差异

| 维度 | 当前架构 | 新架构 |
|------|---------|--------|
| Stage 数量 | 10 个（含 2 终态） | 7 个命名 stage + round 计数器 |
| Stage 命名 | 字符串描述语义 | 同（保持自文档化） |
| 角色 | Worker + Orchestrator | Worker + Orchestrator + **Reviewer** |
| 审查分工 | Orch 做三项 LLM 分析 + 验证 completion | Reviewer 独立验证 commit 是否符合 plan |
| 修正回路 | 无（rejected 为终态，需新建 pipeline） | 有（review_round 计数器，4 轮上限） |
| 表数量 | 3（pipeline_state, context, audit_log） | 4（pipeline, project, register, audit_log） |
| 文件占用 | 无显式机制 | project.文件使用 + agent_status 双向协调 |
| 崩溃恢复 | 三项 analysis_json 断点续跑 | 依赖 Reviewer 定时器 + agent_status 字段 |
| 人工介入 | 无内置机制 | human_intervention 列 + status.py 标红 |
| JSON 载体 | SQLite TEXT 列（同） | SQLite TEXT 列（同） |
| 字段管理 | 预定义列 + ALLOWED_COLUMNS 白名单（同） | 预定义列 + ALLOWED_COLUMNS 白名单（同） |
| CAS + WAL + Trigger | 有 | 有 |
