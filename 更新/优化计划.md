## Phase 1: Pipeline Schema + 核心机制

### 任务 1.1 — 新增 `pipeline_state` 表

单表替代现有 5 表（`lock`, `requests`, `approvals`, `completions`, `processed`）。保留 `context` 表。

```sql
CREATE TABLE IF NOT EXISTS pipeline_state (
    request_id TEXT PRIMARY KEY,
    agent TEXT NOT NULL,
    stage TEXT NOT NULL DEFAULT 'request_submitted' CHECK(stage IN (
        'request_submitted',
        'conflict_analysis_done',
        'boundary_analysis_done',
        'logic_analysis_done',
        'approved',
        'rejected',
        'modifying',
        'self_review_done',
        'completion_submitted',
        'completed',
        'lock_released'
    )),
    revision INTEGER NOT NULL DEFAULT 0,

    -- 请求数据（原 requests 表）
    reason TEXT,
    scope_json TEXT,
    plan_json TEXT,
    self_review_json TEXT,
    constraints_json TEXT,

    -- 审批数据（原 approvals 表）
    approval_status TEXT,
    granted_scope_json TEXT,
    rejection_reason TEXT,
    reviewed_by TEXT,

    -- Completion 数据（原 completions 表）
    completed_at TEXT,
    commits_json TEXT,
    sync_notes TEXT,
    context_updates_json TEXT,

    created_at TEXT DEFAULT (datetime('now','localtime')),
    updated_at TEXT DEFAULT (datetime('now','localtime'))
);
```

### 任务 1.2 — 状态转移规则（Python 层 `VALID_TRANSITIONS`）

```python
VALID_TRANSITIONS = {
    'request_submitted':      ['conflict_analysis_done', 'rejected'],
    'conflict_analysis_done': ['boundary_analysis_done'],
    'boundary_analysis_done': ['logic_analysis_done'],
    'logic_analysis_done':    ['approved', 'rejected'],
    'approved':               ['modifying'],
    'modifying':              ['self_review_done'],
    'self_review_done':       ['completion_submitted'],
    'completion_submitted':   ['completed'],
    'completed':              ['lock_released'],
}
```

### 任务 1.3 — SQL 触发器 `tr_stage_transition`

- `NEW.revision` 必须 = `OLD.revision + 1`（CAS）
- `NEW.stage` 必须是 `OLD.stage` 的合法后继
- 违反任一条件 → `RAISE(ABORT)`

### 任务 1.4 — Python 层 `advance_stage()` 方法

```python
def advance_stage(self, request_id, expected_stage, new_stage, **payload):
    # 1. 读取当前 stage + revision
    # 2. 校验 expected_stage 匹配
    # 3. 校验 (expected_stage → new_stage) 在 VALID_TRANSITIONS 中
    # 4. UPDATE pipeline_state SET stage=new_stage, revision=revision+1,
    #    {**payload columns} WHERE request_id=? AND stage=? AND revision=?
    # 5. 受影响行数为 0 → 重试（最多 3 次）
```

### 任务 1.5 — 旧表处理

`init_db()` 不再创建旧 5 表。`migrate()` 不做旧表迁移（破坏性升级，数据库需重建）。

## Phase 2: 重写 orchestrator.py 方法

### 任务 2.1 — 删除的方法

`submit_request`, `get_request_status`, `wait_for_approval`, `get_approval`,
`issue_approval`, `submit_completion`, `verify_completion`,
`acquire_lock`, `release_lock`, `check_lock`, `is_processed`, `mark_processed`

### 任务 2.2 — 新增的方法

| 方法 | 说明 | 调用方 |
|------|------|--------|
| `init_pipeline(agent, reason, scope, plan, self_review, constraints, tz)` | 创建 pipeline，stage='request_submitted' | Worker |
| `advance_stage(request_id, expected_stage, new_stage, **payload)` | CAS 推进 stage，最多重试 3 次 | 双方 |
| `get_pipeline(request_id)` | 查询单条 pipeline 完整状态 | 双方 |
| `get_pending_requests(agent)` | 查某 agent 未完成的 pipeline | Worker 恢复 |
| `get_requests_by_stage(stage)` | 查处于某 stage 的所有请求 | Orchestrator 扫描 |
| `recover_pipeline(agent)` | 按 agent 查未完成 pipeline，返回断点 stage | 双方重启 |

### 任务 2.3 — 保留的方法（微调）

`__init__`, `_connect`, `init_db`, `migrate`, `check_orchestrator_alive`, `try_register`,
`send_heartbeat`, `get_boundaries`, `set_boundaries`, `update_context`, `run_monitor`

`run_monitor()` 内部查询改为 `pipeline_state` 按 stage 过滤。
